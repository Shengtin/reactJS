React Component

Open up App.js and you're looking at a React component:

function App() {
  return (
    // "HTML" code
  )
}

By now you also know that JS functions can be written as arrow functions as well, that means the App component can also be written like so:

const App = () => {
  return (
    // "HTML" code
  ) 
}

The components above are known as Functional Components, because they are defined using functions. Previously they were also known as Stateless Functional Components as they were unable to hold states within them. Since the introduction of React Hooks it is now possible to hold a state within a Stateless Functional Component. We will learn more about hooks shortly.

The other type of component is known as Stateful Component, it is defined by using class:

class App extends React.Component {
  render() {
    return (
      // "HTML" code here
    )
  }
}

It is much more wordy, but it allows us to store states within the component without the use of hooks, and use component lifecycles.

React Hooks are now the preferred way to use state within components as they allow developers to write less code, but more than that, Functional Components get compiled into much less code compared to Stateful Components, this has huge performance benefits and results in a much smaller project filesize.

Component Tree

When developing a React app, the way you structure your app is similar to a family tree.

Your App.js will always be the highest level component. Inside your App.js you may have child components, such as NavBars, Footers and your different pages, such as a Homepage or an Error page. As we go down the family tree, other components will also live within the child components of the App.js component.

For example, your pages may have different sections or your NavBar may contain a modal to hold a Login or Sign Up form.

You can see the infographic below to visualize how a very simple component tree would look like.

You should always develop your React project with this family tree structure in mind. You want to split your app into smaller more manageable components, some of which may even be multi-purpose and reusable.

Breaking down your app into smaller components makes it much easier to debug your code if something goes wrong.
